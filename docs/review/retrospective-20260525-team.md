## 二、根因分析：为什么9个问题会集中爆发？

### 2.1 直接原因链

```
未读代码就运行脚本
→ 意外改了专题日期
→ 用户指出后仓促修复
→ 修复不验证就推送
→ 用户发现"今日内容"不显示
→ 猜原因而不是调试
→ 连续4轮猜错
→ 第5轮才用浏览器控制台找到data[s]bug
→ 期间用户反复指出相同问题
→ 信心崩塌
```

### 2.2 系统性缺陷

| 缺陷 | 表现 | 影响 |
|------|------|------|
| **没有调试习惯** | JS bug改了几次都是"猜"，没有打开浏览器控制台 | 同一问题拖5轮 |
| **验证缺失** | 修改后不curl/不检查就推送 | 每次推送都是盲盒 |
| **模板意识不足** | 晚报没有固定结构模板，每次自由发挥 | 内容结构随生产轮次改变 |
| **代码阅读缺失** | run update-index.py前没读 | 误改专题日期 |
| **依赖记忆不依赖文档** | 5.22做对了的事（好的音频模板、好的结构）没有固化到文档中 | 每次都要重新发现 |

### 2.3 团队协作问题

5.24已经完成了从"僵化分工"到"灵活分工+质量门禁"的架构升级（v4.0→v5.0），但这套架构在5.25没有被真正执行。

**质量门禁的实际执行情况：**

| 门禁层 | 应该做 | 实际做 |
|--------|-------|-------|
| Layer 1 事实核查 | 每个数据点追溯来源 | ✅ 有来源 |
| Layer 2 风格一致性 | 匹配风格样本 | ❌ 晚报未匹配5.22模板 |
| Layer 3 质量检查 | 18项清单逐一过 | ❌ JS bug漏检、模板结构漏检 |
| Layer 4 数据准确性 | 无编造数据 | ✅ 数据有来源 |

Layer 2和Layer 3的失效是当天问题集中的核心原因。

### 行动3：JS代码必须可验证

**所有前端JavaScript在生产中的变更必须通过浏览器验证：**

```
不需要打开浏览器的方案：
  → 理解代码逻辑后再修改（避免 data[s] 类错误）
  → 修改后对照 data 输入，手动推导输出
  → 写 console.log 辅助理解

需要浏览器的方案：
  → 本地打开 index.html 测试
  → Chrome DevTools > Console 执行关键路径
  → 检查有无报错

绝对禁止：
  → "看着好像没问题，先推了让用户确认"
  → 推了不看结果就直接通知用户
```

### 行动4：运行脚本前强制阅读

**任何已有脚本的运行必须：**

```
1. 先读完整脚本（至少理解输入→处理→输出路径）
2. 确认不会意外修改不该碰的数据
3. 有疑问先问，不假设
4. 运行后检查输出是否符合预期

特别是：
  → update-index.py（改变articles.js和数据日期）
  → generate-charts.py（改变docs/charts/内容）
  → gate-check.py（可能阻塞发布）
```

## 六、致团队

今天的问题可以归结为一句话：

**不是能力不行，是流程失效。**

每一个问题单独拿出来都不难解决。JS的 `data[s]` 错误，在浏览器控制台试一下就能发现。标题不对，curl 一下 `articles.js` 就能确认。但因为没有验证意识，所有问题都是用户发现的。

质量门禁不能只是写在文档里——它需要在每一行代码、每一次推送、每一句话被写出之前真正发挥作用。

5.26早报，不允许再出现5.25的任何一个问题。如果同类问题复发，说明本轮复盘整改本身是失败的。
