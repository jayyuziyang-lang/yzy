## 二、逐问题根因分析

### P0: 首页"今日内容"不显示 — 用户反馈5次，修复4轮

**严重性：** 这是本日最恶劣的问题。用户反复反馈（"网站并没有更新进去""网站还是没看见""只在历史文章里""网站源代码有，但是页面不显示""标题还是没变过来"），每次我都做了局部修补但没有真正定位到根因。

**真实根因链条：**

```
第一层：用户看到"暂无今日内容"
  ↓
第二层：renderToday() 执行了，但 renderEditionList() 没渲染出内容
  ↓
第三层：for (const s of sessions) 中 data[s] 永远是 undefined
  ↓
根因：data[s] 中 s 是对象 {key:'morning',...}，JS强制转为 "[object Object]"
      应为 data[s.key]
```

**为什么花了4轮才找到：**
- 第1轮：认为是 CDN 缓存、部署延迟（推卸给外部因素）
- 第2轮：认为是预渲染内容不够完整（加了更多预渲染 HTML）
- 第3轮：认为是 JS 加载顺序问题（调整了 script 位置）
- **第4轮：才真正读了 JS 代码，发现 data[s] 引用错误**

**本应第1轮就做的事：** 在浏览器控制台执行 `ARTICLES_BY_DATE["2026-05-25"]` 看数据是否存在，然后加 `console.log` 跟踪 `renderToday()` 执行路径。**如果第一轮就这么做了，5分钟解决问题。**

**教训：问题持续越久越不能猜，必须上调试工具。**

---

### P1: 音频人格化丢失 — 用户生气

**直接原因：** 初稿脚本写成了"你好，我是阿扬"，结尾没有晚报预告。

**根因链：**
1. WORKFLOW.md 中脚本模板初始时本身就是错的（写的是"你好，我是阿扬"）
2. 生产时没有参照前几日（5.22）的优秀脚本
3. 初稿输出没有过"人格化门禁"

**教训：** 风格样本不光是文章/SVG的，还应该包括**历史优秀音频脚本**的样本。WORKFLOW.md 已修复。

---

### P2: 图表数据脱节 — 基础质量问题

**直接原因：** `generate-charts.py` 使用硬编码的静态数据，不随当日新闻变化。

**根因：** 这个脚本的设计本身就与"每日定制化内容"的需求矛盾。通用的图表工具生成通用的数据，但财经早报需要与当日叙事对齐的图表。

**修复方案（敷衍的）：** 创建 `charts_today.py` 按日编写。但这需要人工在每期生产时手动编写 Python 代码，不可持续。**

**真正的修复方案应该是：** 要么让 `generate-charts.py` 接受当日参数（新闻关键词），要么放弃通用图表脚本，改为在文章撰写阶段自然地将数据可视化作为文章的一部分（而非独立脚本）。

---

### P3: 深度分析专题日期被篡改

**根因：** 运行 `update-index.py` 前没有阅读其代码。该脚本使用 `os.path.getmtime()` 确定专题目录的日期，git checkout / 文件操作会改变 mtime。

**这是"未读先改"的典型例子。** 如果一个脚本会修改你的数据，运行前必须先理解它的逻辑。

---

### P4: 首页标题错误 + 多余描述行

**根因链：**
1. 文章 `<title>` 写成了"扬说财经 · 早报 | 2026.05.25"（泛写），而非文章实际标题
2. `update-index.py` 读取的是 `<title>` 而非 `<h1>`
3. 首页预渲染内容和 JS 渲染都用了这个泛写标题
4. JS 渲染还额外加了一行 `edition-desc`，用户要求删除

**教训：** 文章 `<title>` 必须与 `<h1>` 一致，这在之前的文章中是惯例（5.22/5.21都是这样的），但5.25没有遵守。

## 三、系统性问题：验证闭环断裂

以上9个问题可以分为三类系统性问题：

### 3.1 不改完不验证（最致命）

**生产流程中缺失的关键步骤：**

| 本应做的事 | 实际做的事 |
|-----------|-----------|
| 修改 JS 后 → 在浏览器中测试 → 确认渲染正确 → 推送 | 修改 JS → 直接推送 → 用户验证 |
| 修改标题后 → 查看 articles.js 确认数据正确 → 刷新首页确认显示 → 推送 | 修改标题 → 直接推送 |
| 音频重新合成后 → 自己听一遍确认 → 再推送 | 重新合成 → 直接推送 |

**结果：每次推送都在把验证责任转嫁给用户。**

### 3.2 不全面验证

每次只验证了"当前修的这个 bug"，没有回归测试"修了这个会不会破坏别的"。

- 修复 JS bug → 标题又错了
- 修复标题 → 用户又发现描述行多余
- 修复音频 → 用户发现音频没推到网站

**每次修复都在制造新的反馈循环。**

### 3.3 不承认不确定

多处问题是因为"我以为这样是对的"但没有确认：
- "我以为 data[s] 是对的" — 没有写过测试或用 console.log 验证
- "我以为标题推上去了" — 没有 curl 检查 articles.js 的内容
- "我以为音频更新了" — 没有对比文件大小就推送了

**Karpathy 原则第1条：先思考再编码。这条在本日生产中从未真正执行过。**

## 五、整改措施（强制执行）

### 5.1 部署必须经过验证

**每一次 git push 后，必须执行完整的验证流程，且验证结果要展示给用户（而不只是自己知道）：**

```
验证清单（强制）：
□ curl index.html | grep data-date → 确认 data-date 为当日
□ curl articles.js | grep latest_date → 确认 latest_date 为当日
□ curl article.html → HTTP 200 且 > 10KB
□ curl audio.mp3 | head -c 100 | file - → 确认是 MP3
□ 在浏览器中打开 index.html（或 curl 查看完整渲染后的关键内容）
```

**任何一条不通过 → 不通知用户，先修复。**

### 5.2 JS 变更必须经过浏览器测试

**所有前端 JavaScript 变更，在推送前必须经过以下至少一项验证：**
1. 在本地浏览器打开 index.html 验证渲染结果（推荐）
2. 在 Chrome DevTools Console 中执行关键路径代码确认无报错
3. 不可能用浏览器时，加 `console.log` 输出然后推送到 GitHub Pages 看控制台输出

**禁止在未验证 JS 行为的情况下推送前端变更。**

### 5.3 运行脚本前必须读代码

运行任何已有脚本（尤其是 `update-index.py`、`generate-charts.py`）前：
1. 阅读完整脚本（至少理解输入→处理→输出路径）
2. 确认不会意外修改不该碰的数据
3. 有疑问先问用户，不假设

### 5.4 文章 <title> 必须与 <h1> 一致

**这不是建议，是规则。** 所有历史文章（5.21、5.22）都遵守了这条规则。5.25 是唯一的例外。

生产流程中增加检查项：
```
□ <title>与<h1>标题一致（或至少包含文章核心标题而非通用站名）
```

### 5.5 用户已指出两次的问题不允许第三次出现

**"用户连续2次指出同一方向错误 → 停下来追问"** — 这条铁律在今天的生产中从未被触达。

具体来说：
- 用户第2次说"今日内容没更新"时，就应该停下手头所有工作，专心调试这个 JS 渲染问题
- 而不是一边修其他问题一边猜
- 用户第2次质疑标题时，就应该对比其他文章的 `<title>` 写法

本复盘本身添加一条新铁律：
**铁律8：用户同一个问题反馈≥2次 → 冻结其他工作，专攻此问题直到解决。不允许带病推送。**
