## 根因分析

### P1: 音频文件不完整（最严重）

**症状：** `audio.mp3` 仅 4MB/1.7分钟，实际脚本 3,079 个中文字符（朗读应~12分钟）。文件被截断或TTS生成不完整。

**根因：**
- 音频生成没有一个明确的验证步骤——生成后无人检查文件大小/时长
- `deploy.sh` 和 `pre-deploy-check.sh` 中没有检查音频文件最小时长的规则
- 生成时的TTS语速参数设置不合理（默认速度导致输出只有1.7分钟）

**修复：**
- 已重新生成：`zh-CN-YunyangNeural` @ -50% 语速，8.3MB / 11.6分钟 ✅
- 文章标注"约12分钟"与实际时长匹配 ✅

### P2: 资源完整性检查缺失

**症状：** 所有18篇文章的HTML中，音频播放器和漫画图片都是硬编码的，没有任何加载失败处理。如果CDN上某个资源缺失，用户看到的是破损的播放器和裂开的图片。

**根因：**
- 文章HTML生成时只关注"内容正确"，没考虑"资源加载失败"场景
- 模板（article-template.html）长期未更新JS逻辑
- 生产流程中缺少"客户端降级体验"的考量

**修复：**
- 所有18篇文章+模板已添加JS：音频8秒超时/加载失败自动隐藏播放器 ✅
- 所有18篇文章+模板已添加JS：漫画SVG失败显示友好占位 ✅
- 首页新增动态媒体指示器：renderEditionList + renderSpecial 均从元数据渲染 ✅

### P3: 资产元数据采集缺失

**症状：** `update-index.py` 不扫描 `audio.mp3` 和 `comic/*.svg`，导致 `data/articles.js` 缺少 `has_audio` / `has_comic` / `comic_count` 字段。首页只能硬编码"图文+音频"。

**根因：**
- update-index.py 当初只采集文章标题/日期/URL，没有扩展资产维度
- 首页JS无法动态判断文章有哪些媒体资源

**修复：**
- update-index.py 新增 `scan_audio_meta()` 和 `scan_comic_meta()` ✅
- 数据层可精确到"音频+7张漫画"级别 ✅

### P4: L3推理验证缺失（新增危机级）

**症状：** 关于"伊朗冲突停火对黄金的影响"的分析判断，在4处内容中出现方向性错误（2篇日报文章 + 深度专题正文 + 音频脚本）。所有数据点都正确（金价$4,502、PCE 3.8%、原油$120），但因果推理的结论完全错误——把"停火"误判为"利好黄金"。

**根因：**
- 分析过程只追踪了一条传导链（通胀→加息→金价承压→停火解除→利好），忽略了主要矛盾方向（避险溢价消退→利空）
- 没有做价格位置自检：黄金在$4,500+（ATH附近），"利好"叙事大概率已被定价
- 没有做历史类比验证：2022年俄乌冲突停火谈判后，金价3个月内从$2,070跌至$1,620
- 没有寻找反方观点：没有任何一个权威来源支撑"停火利好黄金"的论断
- **更深层原因：质量门禁体系只检查了"数据是否准确"（L1），没有检查"推理是否正确"（L3）**

**修复：**
- 已修正4处错误分析 ✅
- 深度专题音频脚本已同步修正 ✅
- 新增Layer 5: 核心逻辑验证门禁（详见 INFO_VERIFICATION_SYSTEM.md）✅

## 整改措施

### S1: 音频生成质量门禁（新增）

在 `pre-deploy-check.sh` 或 `scripts/site-health.py` 中新增：

```bash
# 音频完整性检查
for audio in special/*/audio.mp3; do
    size=$(stat -c%s "$audio")
    if [ "$size" -lt 5000000 ]; then  # 深度专题音频必须>5MB
        echo "❌ $audio 文件过小（${size}B），可能不完整"
        exit 1
    fi
done
```

或者 Python 侧：在 update-index.py 中根据 `audio_size_bytes` 估算时长，低于阈值报警。

**阈值建议：**
- 深度专题：音频 ≥ 5MB（约 8+ 分钟）
- 晚报：音频 ≥ 1.5MB（约 3+ 分钟）
- 早报：音频 ≥ 0.5MB（约 1+ 分钟）

### S2: 统一模板JS升级

article-template.html 已作为**统一JS源**，规则更新步骤：

1. 修改 `special/.templates/article-template.html` 尾部 `<script>` 块
2. 对新专题：直接基于模板生成
3. 对已发布文章：编写批量更新脚本统一同步

**本次批量更新的模式**（后续可直接复用）：

```bash
# 扫描所有文章尾部 <script> 块，替换为最新模板JS
python3 scripts/sync-article-js.py
```

建议新建 `scripts/sync-article-js.py`，实现"一次修改模板，一键同步所有已发布文章"。

### S3: 生产自检清单（深度专题专用）

深度专题作为王牌项目，发布前必须逐项确认：

```
□ 音频文件时长 ≥ 脚本预估时长的80%
□ 漫画SVG全部存在（共X张，路径逐一验证）
□ 阅读时间与音频时间一致（偏差 < 20%）
□ 首页数据已刷新（data/articles.js 含最新元数据）
□ 返回首页链接路径正确
□ 模板JS已同步（audio/comic error handling）
□ 链接到文章的实际路径可访问
```

将此清单加入 `docs/workflow/WORKFLOW.md` 的专题发布流程中。

### S5: 核心逻辑验证门禁（新增）

创建 `docs/team/INFO_VERIFICATION_SYSTEM.md`，建立三层验证体系：

| 层级 | 检查内容 | 验证方法 |
|------|---------|---------|
| L1 数据检查 | 每个数据点正确且有来源 | 来源URL + 等级标注（A/B/C） |
| L2 归因检查 | 因果关系真实成立 | 排除相关性谬误，确认因果方向 |
| L3 推理检查 | 从归因到结论的推演自洽 | 方向性检验 + 历史类比 + 多渠道分析 + 反方观点 |

**强制规则：**
- 所有市场分析/预测类内容，发布前必须通过L3验证
- ATH附近的资产：对"利好"叙述高度警惕
- 没有反方观点支撑的判断，不允许发表
- Layer 5 作为不可绕过的质量门禁加入 AGENT_HANDBOOK.md

## 团队要求

1. **深度专题是王牌产品**，本周暴露的7个问题中，P1（音频不完整）和P2（无资源完整性检查）属于**不可接受**级别。用户打开页面发现音频只有2分钟、漫画加载失败——这直接损害品牌信任。
2. **模板是一切的基础**。article-template.html 的质量决定了所有深度专题的质量。每次模板修改后，必须同步到已发布文章。
3. **"标注什么就是什么"**。标"约12分钟"就必须是10-14分钟，不是1.7分钟。数据精确是信任的起点。
4. **每次发布前运行自检清单**，深度专题不赶时间——质量优先。
5. **信息准确性是生存底线。** 数据正确不等于分析正确。每次发布包含市场分析/预测的内容（包括日报），必须通过L3推理验证——方向性检验、历史类比、多渠道分析、反方观点检查，缺一不可。
6. **同一错误出现在4处内容**，说明问题不在单一编辑环节，而在流程体系。质量门禁必须覆盖从"数据"到"结论"的全程，不能只卡数据不卡逻辑。
7. **用户信任是不可逆的。** 一次错误可能不会立刻失去用户，但持续错误会让品牌贬值。如果用户连续2次指出同一方向错误，停下来追问，不猜测。
