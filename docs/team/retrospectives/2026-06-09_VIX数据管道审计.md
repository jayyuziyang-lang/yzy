## 根因分析

### 根因1：FALLBACK_DATA硬编码单一时点值

`fetch-vix-data.py`中`FALLBACK_DATA`的`current: 21.51`是一个"冻结的快照"——它记录的是6月5日收盘的VIX。只要yfinance被限流，脚本就无条件使用这个值，不管过了多少天。

```python
FALLBACK_DATA = {
    'current': 21.51,  # ← 过了一天还是21.51，不会自动推移
    ...
}
```

**整改：** FALLBACK_DATA必须随最新交易日更新，不能长期停留在某一日的数值。

### 根因2：数据获取缺乏"降级回退"策略

`fetch-vix-data.py`的获取策略是"yfinance → 搜索验证 → 备用数据"，但"搜索验证"这一步的可靠性不稳定，导致频繁回退到备用数据。而备用数据本身没有时效性保障。

**整改：** 增加fallback机制：
- 第一优先：investing.com（手动维护、最权威）
- 第二优先：yfinance实时数据
- 第三优先：WebSearch交叉验证
- 第四优先：本地缓存（记录上次成功获取的值+日期，过期则标记）

### 根因3：手动修改与自动化管道冲突

deploy.sh和deploy.yml都在部署时重新运行fetch-vix-data.py。这意味着：
- 任何对vix-data.js的直接修改 → 部署时被覆盖
- 修复方法只能是修改脚本本身（FALLBACK_DATA），而不是修改输出文件

**整改：** 这是一个系统性问题——手动修正的输出文件不应被自动化管道覆盖，或者管道应支持手动数据覆盖标记。

### 根因4：investing.com不可访问

investing.com是国内用户最常用的VIX数据源之一，但Cloudflare防护使其无法被程序化访问。这使得"以investing.com为权威源"的要求与当前的自动化数据获取能力之间存在根本性矛盾。

**整改：** 
- 短期：通过WebSearch间接获取investing.com页面摘要中的VIX数据
- 中期：建立手动VIX更新SOP（用户代为查询investing.com → 填入脚本）

## 整改措施

### 措施A：更新FALLBACK_DATA为最新值（已执行）

已在`fetch-vix-data.py`中将FALLBACK_DATA更新为VIX 19.92（6月8日收盘），并新增了6月8日的历史数据点。同时更新了`vix-data.js`。

### 措施B：修复手动修改被覆盖问题（新增）

在`fetch-vix-data.py`中添加检查逻辑：**如果vix-data.js已存在且为手动标记（非自动生成），则跳过自动覆盖。**

具体方案：
- vix-data.js头部注释中保留`生成时间`字段
- 脚本运行时检查该时间是否晚于当前执行时间 → 如果是，说明有人手动修改过，跳过写入
- 或者增加一个`__VIX_OVERRIDE__`标记文件

### 措施C：新增"权威源缺失"应急SOP

在WORKFLOW.md中新增：

```
### Phase 3.6: VIX数据更新（2026-06-09复盘追加）

VIX数据更新流程：
1. 首选：运行 fetch-vix-data.py（自动获取）
2. 如果yfinance限流 → 手动搜索"CBOE VIX 最新"交叉验证
3. 如果investing.com结果不可得 → 使用CBOE/Barron's/MarketScreener等替代源
4. 如果必须手动修改 → 必须同时修改 fetch-vix-data.py 的 FALLBACK_DATA
   **不要只改 vix-data.js！** 部署时会被覆盖。
```

### 措施D：长期 — 评估是否可替代investing.com

investing.com的Cloudflare防护短期内无法绕过。建议：
- **主要数据源切换到CBOE官方**（https://www.cboe.com/tradable_products/vix/vix_futures/），虽然同样有防护但至少是VIX的原始发布者
- 或者使用**Alpha Vantage / Tiingo**等有API Key的金融数据服务
- 或通过WebSearch定期轮询多家来源交叉验证

## 经验教训总结

| 教训 | 对应措施 |
|------|---------|
| 手动修改输出文件会被pipeline覆盖 | 修改必须作用在脚本的fallback数据层，而非输出文件 |
| FALLBACK_DATA是"冻结快照"，不会自动推移 | 每次交易日结束后更新FALLBACK_DATA |
| invest.com有Cloudflare防护，程序化不可达 | 建立替代数据源获取方案 |
| VIX数据获取只有一个实时源（yfinance），单点故障 | 至少保留2个独立获取渠道 |
