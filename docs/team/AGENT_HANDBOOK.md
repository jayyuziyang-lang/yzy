## 质量门禁详解

### Layer 1: 事实核查门禁

**谁执行：** Codex 或 Claude Code（灵活）
**谁终审：** Claude Code（不可替代）

**核查标准：**
- 每个百分比数字 → 来源URL
- 每个价格数字 → 具体市场+时间
- 每个历史对比 → 精确参照年份
- 英文公司名 → 首次出现中英对照

**触发阻塞：**
- 发现编造数据 → 整篇退回，根因分析
- 关键数据无来源 → 删除该数据点
- 来源为模型"内部知识" → 不合规

### Layer 1.5: 数据可视化质量门禁（新增）

**谁执行：** Claude Code（不可替代）
**检查项：**
- 运行 `python scripts/generate-charts.py` → 质量门禁 33/33 通过
- 红涨绿跌颜色正确：正→#DC2626，负→#16A34A
- 红流入绿流出颜色正确
- 标签无重叠，正值在柱右侧，负值在柱左侧
- 品牌背景色 #F8FAFC，白边框，淡网格
- 嵌入图表与文章叙事直接相关

**失败 → 不发布，返回修改。详见 WORKFLOW.md Phase 5。**

### Layer 2: 风格一致性门禁

**谁执行：** Claude Code（不可替代）
**检查项：**
- 配色方案匹配风格样本
- 字体/字号/间距一致
- 6故事分类色值正确（红绿紫蓝金银橙）
- 漫画素描风格（线稿为主，色彩克制）

### Layer 3: 质量检查门禁

**谁执行：** Claude Code（不可替代）
**检查清单详见底部。**

### Layer 4: 数据准确性门禁

**谁执行：** Claude Code（不可替代）
**检查项：**
- "历史第X次""史上最大""创X年最大"等表述 → 停，必须有精确来源
- 所有数据可回溯到来源
- 无模型内部知识替代实时搜索

### Layer 5: 核心逻辑验证门禁（新增—2026-05-31复盘追加）

**触发事件：** 伊朗冲突对黄金影响的分析出现系统性L3级推理错误，同一错误出现在4处内容中。数据都正确，但推论全错。

**核心原则：** 数据正确 ≠ 分析正确。每项"因为A所以B"的论断必须通过三层验证（详见 [INFO_VERIFICATION_SYSTEM.md](INFO_VERIFICATION_SYSTEM.md)）。

**谁执行：** Claude Code（不可替代）
**检查项：**
- L1 数据检查：每个支撑数据点有正确的来源
- L2 归因检查：A→B的因果关系是否真实成立（而非相关性）
- L3 推理检查：从归因到结论的推演是否自洽
- 方向性检验：资产在ATH还是底部？利好是否已被定价？
- 历史类比检验：同类历史事件的后续发展是否支持该判断？
- 反方观点检查：是否有至少一个独立来源的相反观点？

**失败 → 不发布。需修改分析逻辑后重新审核。**

## 写作质量标准

### 每条故事的黄金结构

```
[数据] → 发生了什么（精确数字+时间背景）
[叙事] → 为什么会这样（逻辑推演+上下文）
[洞察] → 意味着什么（有态度的判断）
```

### 具体规范

1. **数据精确**：小数点后至少2位，来源可查
2. **有因果**：不报流水账，解释为什么涨/跌
3. **有态度**：每条故事必须有阿扬的判断，不是中立总结
4. **禁止套话**：不写"值得关注""后市可期""需警惕"
5. **禁止编造历史对比**：不写"历史第X次""创X年最大"——除非有精确来源
6. **公司名**：首次出现中英对照（如"超威半导体AMD"），后续只用中文
7. **禁止使用模型内部知识**：每个数据点必须来自实时搜索

### 品质升级 v2.0（2026-06-01 追加 — 对标行业标杆）

**背景：** 对比专业金融简报后，发现我们的早报在数据密度、因果链深度、情景分析三个维度存在系统性差距。详见 `retrospectives/2026-06-01_quality_benchmark.md`。

**强制升级项：**

#### 分析标准
- **传导链≥3步**：每项分析必须包含"事件→中间变量→资产影响→结论"的完整链条
- **双情景框架**：不确定性事项必须给出路径A/路径B两种可能，各带完整传导链
- **核心矛盾总结**：每节末端必须有"核心矛盾未变"或"逻辑链"总结段
- **有态度判断**：不只是"可能涨也可能跌"，而要给出倾向性判断（"在XX条件下，XX方向"）

#### 信息密度标准
- 每节数据点≥5个精确数字
- 有场景细节（如"426艘油轮"而非"大量油轮"）
- 有直接引语（原话引用人物或机构表态）
- 有机构观点引用（摩根士丹利、BofA、高盛等）

#### 搜索流程升级（强制4轮）
```
第1轮：WebSearch("[资产/事件] 最新价格 精确点位")
第2轮：WebSearch("[事件] 最新进展 细节 引语")
第3轮：WebSearch("[事件] 投行 分析 展望")
第4轮：WebSearch("[事件] 历史对比 类比")
```
未完成4轮搜索不得开始写稿。

#### 早报v2.0结构模板
```
□ 行情总览表（精确收盘价+涨跌幅）
□ 头号焦点 🔥（最深度的单事件分析：因果链+双情景）
□ 资产深度 🥇/🛢️（黄金+原油或多资产联动分析）
□ 科技/美股 💻（财报+关键公司数据+机构观点）
□ 宏观政策 🏦（数据解读+央行逻辑链）
□ 本周日历 📅（完整事件时间表）
□ 逻辑框架 🧠（路径A/路径B/独立主线决策树）
```

### 数据准确性铁律

**P0级别违规——发现即阻塞发布：**
- "史上最大""历史第X次""创N年最大"等表述 → 必须验证到精确来源
- 任何百分比/价格数字无来源 → 删除
- 音频/阅读时间标注与实际不符 → 修正

### 新闻时效性铁律（新增 — 2026-05-27）

**P0级别违规：**
- 事件日期错误（把旧闻写成当天新闻）→ 整篇重做
- 重大事件（财报、政策、突破）未标注精确日期 → 补充再发布
- 脚本时间标注与文章不一致 → 修正

**根因分析（5.27英伟达财报日期事件）：**
- 直接原因：使用模型内部知识"5月27日盘后"代替实时搜索
- 根因：缺乏"事件日期必须搜索核实"的强制流程
- 修复：audit-article.py 增加事件时间标注检查 + 铁律写入

**每日必须执行：**
```bash
python scripts/audit-article.py --date YYYY-MM-DD --edition evening
```
- 审核不通过 → 不进入质量终审 → 不发布

## 漫画质量标准（核心特色部门）

### 核心理念

漫画是扬说财经的**核心特色**，但核心目的是**辅助理解**，不是炫技。每张漫画必须有明确的信息传递目标——解释难懂的逻辑、突出重要数据、让读者一眼看懂发生了什么。

**漫画数量不固定，根据需要配图。** 聚焦于关键热点和复杂逻辑，不为了凑数而画。

### 风格基准：双轨制

我们维护**两套风格基准**，分别用于日常版和专题版：

| 场景 | 参照文件 | 定位 | 核心特征 |
|------|---------|------|---------|
| **日常版**（早报/晚报） | `2026-05-22/finance_daily_20260522.html` | 快速信息传递 | 简洁线稿，几何图形，数据驱动 |
| **专题/深度版** | **`special/央行法修订/comic/panel-*.svg`** | 深度叙事理解 | **建筑学隐喻，金色+深蓝配色，叙事驱动，每图独立成篇** |

**日常版风格参照：** `2026-05-22/finance_daily_20260522.html` — 此文的SVG为本团队日常漫画的标准风格。每次生产日常内容前必须先读此参考。

**专题/深度版风格参照：** `special/央行法修订/comic/panel-*.svg` — 此系列的4张SVG为专题漫画的标杆。用户已确认"在这个基础上做的更好"。专题漫画应遵循：
- 每张图有明确的视觉隐喻（柱基/门/水流/对比等建筑学意象）
- 配色体系：暖白底 `#FAFAFA` + 深蓝 `#1E3A5F` + 金色 `#D4A017` + 灰色辅助
- 每张图独立成篇，讲述一个完整概念
- 文字排版讲究，多行对齐，信息分层
- **每次生产专题内容前必须先读此参考。**

### 底线风格规范（两组统一，不可擅自更改）

**风格参数（强制）：**

| 维度 | 标准 | 说明 |
|------|------|------|
| 背景色 | `#faf8f5`（暖白） | 不渐变，无纹理 |
| 线条色 | `#1a1a1a`（近黑） | 2-3px线宽，无阴影 |
| 金色 | `#c9a96e` | 用于强调/趋势线/关键数值 |
| 红色 | `#c62828` | 用于下跌/风险/警示 |
| 绿色 | `#2e7d32` | 用于上涨/利好 |
| viewBox | `300 220` | 固定尺寸，不缩放 |
| 字体 | `PingFang SC, Microsoft YaHei` | sans-serif |
| 人物/角色 | **不加入** | 漫画不上人物形象 |
| 装饰效果 | **无** | 无渐变、无投影、无filter、无发光 |
| 文件大小 | < 5KB | 保持简洁 |

**文字排版规则（强制）：**
- 文字必须放在空白区域，不得与图形重叠
- 重要数字用大号粗体（11-16px），说明用小号（7-8px）
- 上下左右保留充足边距

**任何风格变更（背景色、尺寸、配色、新增元素）必须先展示预览，用户确认后才能发布。**

### 漫画生产流程

```
1. 分析文章 → 确定哪些板块需要漫画（非必需不画）
2. 确定版型 → 日常版参照5.22日报 / 专题版参照央行法修订系列
3. 设计视觉隐喻 → 日常用几何图形+数据 / 专题用建筑学隐喻+金色深蓝色系
4. 编写SVG → 严格按对应风格基准
5. 质量检查 → 文字遮挡/风格匹配/数据准确/文件<5KB
```

### SVG禁止事项

- ❌ 禁止加入任何人物形象/角色/头像
- ❌ 禁止使用深色/渐变背景
- ❌ 禁止使用投影/发光等装饰滤镜
- ❌ 禁止强行加入品牌标识/水印
- ❌ 禁止文字与图形重叠
- ❌ 禁止擅自改变配色方案

## 音频制作规范

### 核心原则

音频必须与文稿完全匹配。"标多少就做多少"——标注"约12分钟"就不能只有1.7分钟。

### edge-tts 标准化参数

| 场景 | TTS语音 | 语速 | 预期时长 | 材料来源 |
|------|---------|------|---------|---------|
| 深度专题 | zh-CN-YunyangNeural | -5% | ~8-12分钟 | script.txt |
| 晚报（日常） | zh-CN-XiaoxiaoNeural | -20% | ~6分钟 | evening/script.txt |
| 晚报（周五/重大事件） | zh-CN-YunyangNeural | -30% | ~9-12分钟 | evening/script.txt |
| 早报 | zh-CN-YunyangNeural | -5% | ~2-3分钟 | morning/script.txt |

### 生成命令示例

```bash
python -m edge_tts \
  --voice zh-CN-YunyangNeural \
  --rate=-5% \
  -f special/<专题目录>/script.txt \
  --write-media special/<专题目录>/audio.mp3
```

### 口播叙事质量标准 v2.0（2026-06-01 升级）

**适用范围：** 早报、晚报、深度专题 — 统一执行，不分版次。

**核心原则：** 不是念新闻，是讲逻辑。音频不是文字的配音版，是另一种叙事形式。

**六条质量标准：**

1. **必须有叙事线索（Thread）**
   - ❌ 坏：先讲A股、再讲美股、再讲黄金……逐条念完
   - ✅ 好：今天三股力量在碰撞——AI、宏观、地缘——它们各自怎么走、之间如何串联
   - 每条线有"假设→证据→判断"三段结构

2. **必须有明确观点（Take）**
   - ❌ 坏："PMI降至50%，生产52.2%，新订单49.8%……"
   - ✅ 好："PMI踩在荣枯线上，工厂还在开工，但客户已经在减少下单了——滞胀这个最棘手的形态正在从担忧变成现实。"
   - 每条信息都要问"所以呢？"——回答就是你的观点

3. **数字必须服务于论证，而非堆砌**
   - ❌ 坏："标普500 7580点涨0.22%，纳指26972涨0.2%，道指51032涨0.72%……"
   - ✅ 好："上周最炸裂的不是指数，是戴尔——单日暴涨33%，AI服务器收入161亿，暴增757%。戴尔不是芯片公司，是组装电脑的。当组装电脑的公司业绩都暴成这样，说明整个AI基础设施链都在受益。"
   - 只讲"支撑判断"的数字，不讲"凑齐数据"的数字

4. **开头3秒决定是否继续听**
   - ❌ 坏："朋友们早上好，先看数据。美股上周五续创新高……"
   - ✅ 好："今天六月一号。三股力量在同一天、从不同方向，同时涌入了市场。"
   - 开头就要告诉听众：今天有大事，而且我帮你理清楚了。

5. **结尾必须有"下注决议"**
   - ❌ 坏："今天的早报就到这里，晚上见。"
   - ✅ 好："三条线的逻辑是串联的。油价能不能降下来，决定了通胀能不能降温，决定了美联储还有没有降息的空间。谁站在中间，谁就要做选择。"
   - 让听众带着一个可以判断的框架离开，而不是一堆散落的事实。

6. **长约3分钟，密度为王**
   - 目标：600-800字（中文，220字/分钟为标准）
   - 超时说明你没想清楚核心论点
   - 不靠堆时间凑"工作量感"，靠密度和洞察建立"价值感"
   - 早报严格控制在2.5-3.5分钟，晚报4-6分钟，深度专题8-15分钟

**验证方法（自查清单）：**
- [ ] 读一遍：开头3秒有没有悬念？
- [ ] 读一遍：有没有"所以呢"的空信息？
- [ ] 读一遍：去掉所有数字还剩什么——还有观点吗？
- [ ] 读一遍：结尾能让人记住一句话吗？

## 数据可视化（Python工具链）

### 核心理念

Python图表工具（`scripts/generate-charts.py` + `scripts/chart_templates.py`）是我们已有的强大资产。数据图表 ≠ 漫画，两者互补：
- **漫画**：解释逻辑、视觉隐喻、让读者"看懂"发生了什么
- **数据图表**：展示真实数据走势、让读者"看清"数据的全貌

**两者在文章中同时使用，不互相替代。**

### 品牌风格系统 v3.0（2026-05-27 专业财经媒体升级）

所有图表现在统一使用以下品牌设计系统：

**v4.0 核心升级 — ChartCard 专业布局引擎：**
- 卡片式设计：暖灰背景（`#F0F2F5`）+ 纯白图表面板（`#FFFFFF`）
- 专业标题层级：**主标题（粗体14pt）→ 副标题（灰9pt）→ 图表面板 → 来源脚注（灰6.5pt）**
- 零脊线（所有 spines 移除），仅保留水平网格线（置于数据下层）
- 设计参考：The Economist / Financial Times 混合风格

| 组件 | 文件 | 用途 |
|------|------|------|
| 品牌mplstyle v2.0 | `stylelib/yangshuo.mplstyle` | 全局 rcParams（暖灰背景/纯白卡片/零脊线/浅灰网格） |

| 图表布局引擎 | `generate-charts.py` → `ChartCard` 类 | 统一标题/副标题/来源脚注布局 |
| 高级模板库 | `scripts/chart_templates.py` | 6种进阶图表（棒棒糖/瀑布/斜率/哑铃/华夫饼/置信区间） |

**每张图表的黄金结构（ChartCard 自动执行）：**

## 部署规范：deploy.sh是唯一通道

### 铁律：禁止手动 git push

**背景：** 5.25当天反复出现"改完HTML忘记run update-index.py"、"改完数据手动git push不验证"等部署流程遗漏。原因是我们一直用 `git add → git commit → git push` 手动操作，跳过了所有验证环节。

**规则：**
- 所有部署必须使用 `bash deploy.sh "提交信息"`
- `deploy.sh` 自动执行：`update-index.py` → `pre-deploy-check.sh`(7项验证，含图表质量门禁) → `git push` → CDN预热 → 部署后验证
- 手动 `git add . + git commit + git push` 被 pre-push 钩子拦截
- 特殊情况需强制推送：`GIT_PUSH_USING_DEPLOY=1 git push`

### pre-push 钩子安装

初次使用执行：
```bash
bash scripts/install-hooks.sh
# 或手动复制：
cp scripts/pre-push.sh .git/hooks/pre-push
chmod +x .git/hooks/pre-push
```

### 部署前验证内容（8项）

`pre-deploy-check.sh` 自动检查：
1. ✅ 运行 `update-index.py` 刷新数据索引
2. ✅ 验证 `articles.js` 包含最新日期
3. ✅ 验证 `index.html` 预渲染包含晚报链接
4. ✅ 验证所有文章"返回首页"链接为 `../../../index.html`
5. ✅ 检查无模板占位符残留
6. ✅ 检查git状态
7. ✅ 运行 generate-charts.py 图表质量门禁（33项检查，阻塞部署）
8. ✅ **运行 audit-article.py 综合审核（新增 — 时效性/真实性/来源/一致性，阻塞部署）**

其中任何一项失败 → 阻塞部署 → 修复后重新运行。

### 关键修复：标题自动同步（5.25事件根因）

**5.25 bug 的完整根因链：**
1. 文章写完时标题为A → `index.html` 预渲染了标题A
2. 后来标题改为B → 手动更新了文章HTML文件
3. 运行 `update-index.py` → `data/articles.js` 正确读到了标题B
4. 但 `index.html` 的预渲染内容仍为标题A（没人手动改）
5. **JS逻辑错误**：`renderToday()` 检查"预渲染链接数>=预期数" → return 跳过 → **用户看到旧标题**

**双管齐下修复：**

**修复1 — `update-index.py` 自动更新预渲染（2026-05-25部署）**
- `update-index.py` 现在会通过 `<!-- TODAY_PRERENDER_START/END -->` 和 `<!-- HISTORY_PRERENDER_START/END -->` 标记位，自动更新 `index.html` 中的预渲染内容
- 每次运行都会根据最新文章数据生成 todayContent 和 historyContent
- 部署流程变为：`update-index.py`(含预渲染) → `pre-deploy-check.sh`(8项验证，含综合审核) → 部署

**修复2 — `renderToday()` 永远以数据层为准（2026-05-25部署）**
- 移除了"预渲染链接数>=预期数就跳过"的逻辑
- JS 加载后永远用 `ARTICLES_BY_DATE` 数据覆盖预渲染区域
- 预渲染仅用于首屏展示和SEO，JS数据层才是权威来源
