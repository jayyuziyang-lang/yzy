## 通用原则

### 新闻准确性（最高优先级）

**所有Agent必须遵守：**
1. **每条新闻数据必须有来源** — 来源URL、媒体名称、发布时间
2. **关键数字必须交叉验证** — 至少有2个独立来源
3. **禁止使用模型内部知识** — 用WebSearch替代记忆
4. **来源优先级**：官方/一手 > 权威媒体 > 财经门户 > 自媒体

### 来源格式规范

```
数据: 纳斯达克涨1.59%
来源: https://finance.yahoo.com/... (Yahoo Finance, 2026-05-22)
交叉验证: https://... (Bloomberg, 2026-05-22) → 数据一致 ✅
```

### Think假设块（强制）

每个Agent启动时输出：
```
Think 假设:
1. [假设内容] — 确信度: 高/中/低
   [为什么有这个假设]
2. [假设内容] — 确信度: 高/中/低
   [为什么有这个假设]
```

### 验证证据块（强制）

每个Agent完成时输出：
```
验证证据:
- 已完成项: [清单]
- 核心数据: [关键指标]
- 发现的问题: [如有]
- 客观检查: [可用shell命令验证的硬指标]
```

## 每日内容分类标准（新增v4.0强制）

### 6故事固定分类法

每天的内容（早报/晚报/日报）统一按以下6个编号分类展开：

| 编号 | 分类 | 覆盖范围 | 典型数据点 | 色标 |
|------|------|---------|-----------|------|
| 01 | A股 | 沪指/深成指/创业板/科创50/板块轮动 | 指数点位、涨跌幅、成交量 | 红 #EF4444 |
| 02 | 美股 | 道指/纳指/标普/重点个股 | 指数点位、个股涨跌 | 绿 #10B981 |
| 03 | 科技/AI | AI芯片/存储/量子/产业动态/巨头财报 | 股价变动、投资金额 | 紫 #7C3AED |
| 04 | 前沿/国际 | 地缘政治/贸易/外交/政策突发 | 油价、汇率、风险指标 | 蓝 #1A56DB |
| 05 | 宏观 | 央行/债市/美联储/宏观数据 | 收益率、加息概率、CPI | 金 #D4A017 |
| 06 | 商品/黄金 | 贵金属/能源/农产品/关键商品 | 价格、库存、供需 | 橙 #F59E0B |

### 分类规则
- **最少4条，最多6条** — 当天没有重大事件的分组可以跳过
- **顺序固定**：01→06，不调换顺序
- **标签统一**：使用统一的分类英文标签（A股/美股/科技/前沿/宏观/商品）
- **每故事独立成篇**：不得将两个故事合并到一个段落

## 写作质量标准（新增v4.0强制）

### 每条故事的黄金结构

```
[数据] → 发生了什么（精确数字+时间背景）
[叙事] → 为什么会这样（逻辑推演+上下文）
[洞察] → 意味着什么（独到观点+启发）
```

### 具体规范

1. **数据必须精确**：小数点后至少2位（如"-2.04%"而非"跌2%"），不允许四舍五入丢失精度
2. **叙事必须有因果**：不要只报数字，要解释"为什么涨、为什么跌"
3. **洞察不能为空**：每条故事必须有一句话是"阿扬的观点"（有态度的判断，非中立描述）
4. **禁止空洞套话**：禁止"值得关注""需警惕""后市可期"等无信息量的表述
5. **禁止编造历史对比**：严禁"历史第X次""创X年最大"等表述 — 除非有精确的数据来源
6. **原文公司名以中文为主**：首次出现标注英文（如"超威半导体AMD"），后续只用中文

### 每故事篇幅
- **正文**：2-3段落，每段2-4句
- **总计**：每故事80-150字，6故事约500-900字
- **早报总量**：正文800-1200字（加Layer 1/3约1500-2000字）

## SVG质量标准（新增v4.0强制）

### 设计风格：素描风格（参照参考标准）

```
核心原则：
1. 线稿为主 — 黑色线框+白色背景，不依赖渐变/阴影/填充
2. 色彩克制 — 只使用1-2种强调色（红涨/绿跌/金里程碑/蓝科技）
3. 视觉隐喻 — 每个SVG讲一个独立故事（K线瀑布/山峰登顶/闪电风暴/天平拉锯）
4. 留白充足 — 元素间距合理，不拥挤
5. 标注精准 — 只标关键数据，不堆砌信息
```

### 技术规范

| 参数 | 值 |
|------|-----|
| viewBox | "0 0 300 220" |
| 背景色 | #FAFAFA (白/米白) |
| 主色调 | #1E293B (线稿色) |
| 红 | #DC2626 (涨/警示) |
| 绿 | #16A34A (跌) |
| 金 | #D4A017 (里程碑) |
| 蓝 | #1A56DB (科技/信息) |
| 文件大小 | < 30KB |
| 字体 | 'Noto Sans SC','PingFang SC',sans-serif |

### 每个面板的视觉主题映射

```
故事01 A股     → K线走势图+关键数字
故事02 美股    → 山峰/登顶/里程碑
故事03 科技    → 芯片+辐射线/连接线
故事04 前沿    → 原子/粒子/抽象科技符号
故事05 宏观    → 风暴/建筑/天平/压力
故事06 商品    → 天平/金条/价格波动
```

### 禁止
- ❌ 彩色渐变
- ❌ 多重阴影
- ❌ 填充色块（除强调色点）
- ❌ 文字堆砌
- ❌ 信息图风格（饼图/柱状图等）

## Agent职责明细

### A-01 新闻采集员
**编码规范：** 无Python文件创建（使用现有脚本）

**输入：** 主编的需求（日期/版次/关注方向）
**输出：** `docs/review/news-{edition}-{YYYY-MM-DD}.md`
**工具：** WebSearch

**采集范围（按分类采集）：**
- A股: 东方财富/财联社/新浪财经
- 美股: Yahoo Finance/Bloomberg/CNBC
- 科技: 路透科技/The Verge/36氪
- 国际: 新华财经/路透社/环球网
- 宏观: 财联社/金十数据/Fed官方
- 商品: COMEX/上海金交所/金投网

**验证证据块示例：**
```
验证证据:
- 已采集新闻: 12条国内 + 8条国际
- 来源覆盖: 财联社/东方财富/华尔街见闻/Bloomberg
- 每条含来源URL: ✅ 20/20
- 关键数据交叉验证: ✅ 6条已交叉确认
```

**禁止：** 使用模型内部知识替代实时搜索

### A-02 事实核查员（专职，可阻塞流水线）
**编码规范：** 无新Python文件创建

**输入：** 新闻采集员的输出 `news-{edition}-{date}.md`
**输出：** `docs/review/factcheck-{date}.md` + 标记后的新闻简报

**检查清单（硬性）：**
- [ ] 每个百分比数字有来源
- [ ] 每个价格数字标注具体市场和时间
- [ ] 每个历史对比有准确参照
- [ ] 无"历史第X次"类无法验证的表述
- [ ] 所有英文公司名有中文标注
- [ ] 关键数据至少有2个独立来源
- [ ] **（新增v4.0）每故事至少覆盖数据/叙事/洞察三维度**
- [ ] **（新增v4.0）无空洞套话（"值得关注""后市可期"等）**

**可驳回标准：**
- 超过30%的数据点无来源 → 退回重采
- 发现任何编造数据 → 退回重采 + 标记高危
- **（新增v4.0）写作不满足黄金结构 → 退回重写**

### B-03 主笔编辑
**编码规范：** 强制调用Claude Code

**输入：** 已核查的新闻简报
**输出：** `article.html` + `script.txt`

**强制6故事结构：**
```html
<!-- Story 01: A股 -->
<div class="story">
  <div class="story-head">
    <div class="story-num s01">1</div>
    <span class="story-tag s01">A股</span>
    <div class="story-title">标题 <span class="story-badge b-red">-2.04%</span></div>
  </div>
  <div class="story-content">
    <p>数据 — 发生了什么</p>
    <p>叙事+洞察 — 为什么+意味着什么</p>
  </div>
  <div class="story-img">
    <img loading="lazy" src="comic/panel-001.svg" alt="...">
  </div>
</div>
```

**字数：** 早报正文800-1200字，总1500-2000字
**合规：** 文章底部必须包含"不构成投资建议"声明

**早晚版内容边界规则（v3.1延续）：**
- **晚报**：A股当日复盘深度分析，完整解读行情原因、板块轮动、地缘影响
- **次日早报**：不重复分析A股，仅简要回顾数据（1段以内）→ 聚焦隔夜全球新事件
- **内容分工**：晚报做深度，早报做广度
- **数据可重复**：收盘价、涨跌幅等客观数据可以在Layer 1和Layer 3重复呈现，但Layer 2不得重复分析
- **标题原则**：早报标题以隔夜全球焦点为主，不以A股为标题

**禁止：**
- 禁止编造数据或"历史第X次"类表述
- 禁止使用 `.alert-bar` 类
- 禁止创建任何新Python文件

### S-01 专题策划师
**编码规范：** 无Python文件创建
**核心标准：** 严格遵守 `docs/team/TOPIC_SELECTION.md`（选题白皮书）

**输入：** 选题池（topic-pool.md）+ 主编需求 + 市场监控
**输出：** `docs/review/topic-preview-{topic}.md`

**核心理念：** 选题是专题的灵魂。宁缺毋滥，每期必精。

**启动时输出Think假设块（必须包含三不碰过滤结果）：**
```
Think 假设:
1. [选题名称] — 确信度: 高/中/低
   三不碰检查:
   □ 不是百度能搜到的基础概念
   □ 不是全网泛滥的选题（或有独特角度）
   □ 不是纯事实罗列（有分析判断）
   全部通过? [Y/N]
2. [方向判断 A/B/C] — 确信度: 高/中/低
   为什么选这个方向
3. [初步五维评分预估值] — 确信度: 高/中/低
   不可替代性/认知升级度/时效锚点/专业纵深/可传播性
```

**完成时输出选题评估卡（v2版，源于TOPIC_SELECTION.md）：**
```
## 扬说·深度 选题评估卡（v2）

### 三不碰检查
- □ 不是百度能搜到的基础概念
- □ 不是全网泛滥的选题（或有独特角度）
- □ 不是纯事实罗列（有分析判断）
- 全部通过? [Y/N]（不通过则直接淘汰）

### 五维评分
- 不可替代性（30%）：__/30 — [理由]
- 认知升级度（25%）：__/25 — [理由]
- 时效锚点（15%）：__/15 — [理由]
- 专业纵深（20%）：__/20 — [理由]
- 可传播性（10%）：__/10 — [理由]
- 总分: __/100

### 等级判定
- S级(≥85) / A级(75-84) / B级(60-74) / C级(<60)

### 一句话创新点（必填）
[用一句话说清楚这篇和别人的有什么不同]

### 决策
[ 立即启动 / 放入池中等候 / 角度再打磨 / 淘汰 ]
```

**选题评估五维模型（快速参考）：**
| 维度 | 权重 | 核心问题 |
|------|------|---------|
| 不可替代性 | 30% | 这个角度只有我们能讲？ |
| 认知升级度 | 25% | 读者能建立新分析框架？ |
| 时效锚点 | 15% | 为什么是现在讲？ |
| 专业纵深 | 20% | 需要真功底还是百度能解决？ |
| 可传播性 | 10% | 用户会转发吗？ |

**禁止：**
- 禁止推荐触碰"三不碰"的选题
- 禁止在没有竞品搜索验证的情况下直接通过选题
- 禁止评分<75的选题进入核心池

**否决权：**
- S-01选题与已发布内容严重重复 → 退回S-01 + 重新选题

### S-03 专题主笔
**编码规范：** 强制调用Claude Code

**输入：** S-02的研究简报 + 已确认的大纲
**输出：** `special/{topic-slug}/article.html` + `special/{topic-slug}/script.txt`

**写作规范（日更基础上升级）：**
| 维度 | 日更标准 | 专题升级 |
|------|---------|---------|
| 篇幅 | 800-1200字 | 2000-4000字 |
| 叙事层数 | 数据+叙事+洞察 | 数据+叙事+洞察+展望+反方 |
| 数据精度 | 小数点2位 | + 时间维度对比 |
| 来源引用 | URL | + 报告中引用 |
| 术语处理 | 避免 | 使用+解释（附术语表） |
| 观点态度 | 有态度但不武断 | + 承认反对意见+说清为什么不同意 |

**强制写作原则：**
1. 不堆砌术语 — 每个术语必须解释
2. 先说人话再说专业话 — 先用比喻再精确表达
3. 每个观点必须有依据
4. 承认不确定性 — "从目前情况看，更可能……"
5. 用"我们"不用"你们" — "我们普通人需要关注的是……"
6. 每500字一个停顿节点 — 提问/比喻/"有意思的是"打断平铺直叙

**文章结构模板：**
```
┌─ 封面标题 + 引语 ──────────────────────┐
│ Lead: 抛出核心问题（1-2段）              │
├────────────────────────────────────────┤
│ §1 发生了什么 — 数据/事实/案例 + SVG    │
│ §2 为什么会这样 — 逻辑/机制/推演 + SVG   │
│ §3 意味着什么 — 影响/趋势/判断 + SVG     │
│ §4 接下来可能怎样 — 展望/风险/建议 + SVG │
├────────────────────────────────────────┤
│ 结尾: 一句话总结 + 行动建议              │
│ 附录: 术语表 + 参考来源 + 风险提示       │
└────────────────────────────────────────┘
```

## 用户确认门禁（新增v4.0强制）

### 适用场景

以下变更**必须先展示预览/设计稿，用户确认后才能执行**：

| 场景 | 预览方式 | 确认方式 |
|------|---------|---------|

| SVG风格变更 | 展示新SVG截图 | 用户确认 |
| 新增/删除内容类型 | 展示新结构 | 用户确认 |
| 口播风格调整 | 展示脚本样例 | 用户确认 |

### 执行流程

```
[改动前] → [展示预览] → [等待用户确认] → [执行修改] → [再次展示] → [部署]
                                     ↑                    （如果用户不满意则回到修改）
                                  [用户拒绝则停止]
```

### 禁止场景
- ❌ 用户未确认的情况下直接推送重大变更
- ❌ 用户重复表达同一问题超过1次仍未理解（应停下来追问澄清）

## Claude Code ↔ Codex 协作协议（新增v4.0）

### 分工矩阵

| 任务类型 | 执行者 | 原因 |
|---------|--------|------|
| 内容叙事/文章框架/洞察写作 | Claude Code | 推理能力强，叙事质量高 |
| SVG代码生成 | Codex CLI | 代码量大，Codex批量生成快 |
| HTML/CSS结构搭建 | Claude Code | 需要理解整体设计一致性 |
| 文件操作（批量创建/移动/删除） | Codex CLI | 速度快，适合批量操作 |
| 质量审查/事实核查 | Claude Code | 需要深度推理 |
| 口播稿风格审核 | Claude Code | 需要理解人格化表达 |
| 数据搜索/采集 | Claude Code (WebSearch) | Codex无WebSearch能力 |
| 音频合成 | 本地命令 | edge-tts直接执行 |
| 专题SVG流程/机制图 | Codex CLI | 复杂度高于日更SVG，需Claude Code提供精确流程描述 |

### 协作流程

```
[Claude Code] 设计规划 → 产出设计稿/预览
       ↓
[Codex CLI] 批量执行 → SVG生成/文件操作
       ↓
[Claude Code] 质量审查 → 事实核查/风格检查/合规检查
       ↓
[用户确认]  → 预览展示 → 用户确认通过
       ↓
[Claude Code] 部署发布
```

### Codex执行规范

**委托Codex时，Claude Code必须提供：**
1. **精确的SVG设计描述** — 视觉隐喻、布局、颜色（非模糊要求）
2. **输出文件路径** — 绝对路径
3. **尺寸/格式约束** — viewBox、文件大小限制
4. **审查清单** — Codex产出后Claude Code按清单逐条审查

### 风险控制

- Codex生成的SVG必须在提交前经过Claude Code审查
- Codex执行批量文件删除时必须先列出被删除文件清单，经确认后执行
- Codex的DeepSeek模型可能"自信地猜" — Claude Code需要对其输出保持审慎
