## 通用原则

### 新闻准确性（最高优先级）

**所有Agent必须遵守：**
1. **每条新闻数据必须有来源** — 来源URL、媒体名称、发布时间
2. **关键数字必须交叉验证** — 至少有2个独立来源
3. **禁止使用模型内部知识** — 用WebSearch替代记忆
4. **来源优先级**：官方/一手 > 权威媒体 > 财经门户 > 自媒体

### 来源格式规范

```
数据: 纳斯达克涨1.59%
来源: https://finance.yahoo.com/... (Yahoo Finance, 2026-05-22)
交叉验证: https://... (Bloomberg, 2026-05-22) → 数据一致 ✅
```

### Think假设块（强制）

每个Agent启动时输出：
```
Think 假设:
1. [假设内容] — 确信度: 高/中/低
   [为什么有这个假设]
2. [假设内容] — 确信度: 高/中/低
   [为什么有这个假设]
```

### 验证证据块（强制）

每个Agent完成时输出：
```
验证证据:
- 已完成项: [清单]
- 核心数据: [关键指标]
- 发现的问题: [如有]
- 客观检查: [可用shell命令验证的硬指标]
```

## 每日内容分类标准（新增v4.0强制）

### 6故事固定分类法

每天的内容（早报/晚报/日报）统一按以下6个编号分类展开：

| 编号 | 分类 | 覆盖范围 | 典型数据点 | 色标 |
|------|------|---------|-----------|------|
| 01 | A股 | 沪指/深成指/创业板/科创50/板块轮动 | 指数点位、涨跌幅、成交量 | 红 #EF4444 |
| 02 | 美股 | 道指/纳指/标普/重点个股 | 指数点位、个股涨跌 | 绿 #10B981 |
| 03 | 科技/AI | AI芯片/存储/量子/产业动态/巨头财报 | 股价变动、投资金额 | 紫 #7C3AED |
| 04 | 前沿/国际 | 地缘政治/贸易/外交/政策突发 | 油价、汇率、风险指标 | 蓝 #1A56DB |
| 05 | 宏观 | 央行/债市/美联储/宏观数据 | 收益率、加息概率、CPI | 金 #D4A017 |
| 06 | 商品/黄金 | 贵金属/能源/农产品/关键商品 | 价格、库存、供需 | 橙 #F59E0B |

### 分类规则
- **最少4条，最多6条** — 当天没有重大事件的分组可以跳过
- **顺序固定**：01→06，不调换顺序
- **标签统一**：使用统一的分类英文标签（A股/美股/科技/前沿/宏观/商品）
- **每故事独立成篇**：不得将两个故事合并到一个段落

## 写作质量标准（新增v4.0强制）

### 每条故事的黄金结构

```
[数据] → 发生了什么（精确数字+时间背景）
[叙事] → 为什么会这样（逻辑推演+上下文）
[洞察] → 意味着什么（独到观点+启发）
```

### 具体规范

1. **数据必须精确**：小数点后至少2位（如"-2.04%"而非"跌2%"），不允许四舍五入丢失精度
2. **叙事必须有因果**：不要只报数字，要解释"为什么涨、为什么跌"
3. **洞察不能为空**：每条故事必须有一句话是"阿扬的观点"（有态度的判断，非中立描述）
4. **禁止空洞套话**：禁止"值得关注""需警惕""后市可期"等无信息量的表述
5. **禁止编造历史对比**：严禁"历史第X次""创X年最大"等表述 — 除非有精确的数据来源
6. **原文公司名以中文为主**：首次出现标注英文（如"超威半导体AMD"），后续只用中文

### 每故事篇幅
- **正文**：2-3段落，每段2-4句
- **总计**：每故事80-150字，6故事约500-900字
- **早报总量**：正文800-1200字（加Layer 1/3约1500-2000字）

## SVG质量标准（新增v4.0强制）

### 设计风格：素描风格（参照参考标准）

```
核心原则：
1. 线稿为主 — 黑色线框+白色背景，不依赖渐变/阴影/填充
2. 色彩克制 — 只使用1-2种强调色（红涨/绿跌/金里程碑/蓝科技）
3. 视觉隐喻 — 每个SVG讲一个独立故事（K线瀑布/山峰登顶/闪电风暴/天平拉锯）
4. 留白充足 — 元素间距合理，不拥挤
5. 标注精准 — 只标关键数据，不堆砌信息
```

### 技术规范

| 参数 | 值 |
|------|-----|
| viewBox | "0 0 300 220" |
| 背景色 | #FAFAFA (白/米白) |
| 主色调 | #1E293B (线稿色) |
| 红 | #DC2626 (涨/警示) |
| 绿 | #16A34A (跌) |
| 金 | #D4A017 (里程碑) |
| 蓝 | #1A56DB (科技/信息) |
| 文件大小 | < 30KB |
| 字体 | 'Noto Sans SC','PingFang SC',sans-serif |

### 每个面板的视觉主题映射

```
故事01 A股     → K线走势图+关键数字
故事02 美股    → 山峰/登顶/里程碑
故事03 科技    → 芯片+辐射线/连接线
故事04 前沿    → 原子/粒子/抽象科技符号
故事05 宏观    → 风暴/建筑/天平/压力
故事06 商品    → 天平/金条/价格波动
```

### 禁止
- ❌ 彩色渐变
- ❌ 多重阴影
- ❌ 填充色块（除强调色点）
- ❌ 文字堆砌
- ❌ 信息图风格（饼图/柱状图等）

## Agent职责明细

### A-01 新闻采集员
**编码规范：** 无Python文件创建（使用现有脚本）

**输入：** 主编的需求（日期/版次/关注方向）
**输出：** `docs/review/news-{edition}-{YYYY-MM-DD}.md`
**工具：** WebSearch

**采集范围（按分类采集）：**
- A股: 东方财富/财联社/新浪财经
- 美股: Yahoo Finance/Bloomberg/CNBC
- 科技: 路透科技/The Verge/36氪
- 国际: 新华财经/路透社/环球网
- 宏观: 财联社/金十数据/Fed官方
- 商品: COMEX/上海金交所/金投网

**验证证据块示例：**
```
验证证据:
- 已采集新闻: 12条国内 + 8条国际
- 来源覆盖: 财联社/东方财富/华尔街见闻/Bloomberg
- 每条含来源URL: ✅ 20/20
- 关键数据交叉验证: ✅ 6条已交叉确认
```

**禁止：** 使用模型内部知识替代实时搜索

### A-02 事实核查员（专职，可阻塞流水线）
**编码规范：** 无新Python文件创建

**输入：** 新闻采集员的输出 `news-{edition}-{date}.md`
**输出：** `docs/review/factcheck-{date}.md` + 标记后的新闻简报

**检查清单（硬性）：**
- [ ] 每个百分比数字有来源
- [ ] 每个价格数字标注具体市场和时间
- [ ] 每个历史对比有准确参照
- [ ] 无"历史第X次"类无法验证的表述
- [ ] 所有英文公司名有中文标注
- [ ] 关键数据至少有2个独立来源
- [ ] **（新增v4.0）每故事至少覆盖数据/叙事/洞察三维度**
- [ ] **（新增v4.0）无空洞套话（"值得关注""后市可期"等）**

**可驳回标准：**
- 超过30%的数据点无来源 → 退回重采
- 发现任何编造数据 → 退回重采 + 标记高危
- **（新增v4.0）写作不满足黄金结构 → 退回重写**

### B-03 主笔编辑
**编码规范：** 强制调用Claude Code

**输入：** 已核查的新闻简报
**输出：** `article.html` + `script.txt`

**强制6故事结构：**
```html
<!-- Story 01: A股 -->
<div class="story">
  <div class="story-head">
    <div class="story-num s01">1</div>
    <span class="story-tag s01">A股</span>
    <div class="story-title">标题 <span class="story-badge b-red">-2.04%</span></div>
  </div>
  <div class="story-content">
    <p>数据 — 发生了什么</p>
    <p>叙事+洞察 — 为什么+意味着什么</p>
  </div>
  <div class="story-img">
    <img loading="lazy" src="comic/panel-001.svg" alt="...">
  </div>
</div>
```

**字数：** 早报正文800-1200字，总1500-2000字
**合规：** 文章底部必须包含"不构成投资建议"声明

**早晚版内容边界规则（v3.1延续）：**
- **晚报**：A股当日复盘深度分析，完整解读行情原因、板块轮动、地缘影响
- **次日早报**：不重复分析A股，仅简要回顾数据（1段以内）→ 聚焦隔夜全球新事件
- **内容分工**：晚报做深度，早报做广度
- **数据可重复**：收盘价、涨跌幅等客观数据可以在Layer 1和Layer 3重复呈现，但Layer 2不得重复分析
- **标题原则**：早报标题以隔夜全球焦点为主，不以A股为标题

**禁止：**
- 禁止编造数据或"历史第X次"类表述
- 禁止使用 `.alert-bar` 类
- 禁止创建任何新Python文件

### B-04 漫画设计师
**设计规范：** 遵循上方 [SVG质量标准](#svg质量标准新增v40强制)

**输入：** 文章内容
**输出：** `comic/panel-001.svg` ~ `panel-006.svg`
**尺寸：** viewBox="0 0 300 220"
**风格：** 素描风格 — 线稿为主，色彩克制，视觉隐喻

**质量检查：**
- [ ] SVG文件大小 < 30KB
- [ ] 线稿风格（非信息图）
- [ ] 色彩不超过3种（含黑色）
- [ ] 视觉隐喻清晰
- [ ] 留白充足，不拥挤
- [ ] 文字在SVG内可选中（非图片）

**委托Codex生成原则：**
- SVG代码量大 → 委托给Codex CLI生成
- Claude Code提供设计方向和视觉隐喻描述
- Claude Code审查SVG是否符合素描风格标准

### B-05 音频制作人
**输入：** article.html + script.txt
**输出：** `script.txt` + `audio.mp3`

**分步执行：**
1. 写口播稿 → `script.txt`（必须遵循6故事结构）
2. 主编审查
3. 审查通过 → 合成 `audio.mp3`

**音频参数：** `python -m edge_tts --voice zh-CN-YunyangNeural --rate=-5%`
**脚本规范：** 纯中文文本，无英文字母、无URL、无XML/HTML标签
**口播风格：** 口语化（"昨天""隔夜那边""有意思的是"），保持阿扬人格化表达

### B-06 数据分析师
**输入：** 文章中的数据需求
**输出：** `docs/charts/*.svg`（调用 `generate-charts.py`）

**流程：**
1. 运行 `python scripts/generate-charts.py`
2. 验证生成的SVG文件存在
3. 将有价值的图表嵌入文章 Layer 3

### C-07 质检审查官（专职，独立门禁）
**编码规范：** 强制调用Claude Code

**输入：** 所有产出文件
**输出：** 质量审查报告

**方法：** 收集客观证据（使用shell命令验证硬指标）

**检查清单（硬性）：**
- [ ] article.html 存在且>10KB
- [ ] 所有 SVG 面板 < 30KB
- [ ] SVG为素描风格（非信息图）
- [ ] 数据表数据完整，每行有涨跌幅
- [ ] 合规声明"不构成投资建议"存在
- [ ] 无"历史第X次"类可疑表述
- [ ] 音频文件存在且>100KB
- [ ] **（新增v4.0）6故事结构完整**
- [ ] **（新增v4.0）每故事含数据+叙事+洞察**
- [ ] **（新增v4.0）无空洞套话**
- [ ] **（新增v4.0）口播稿与文章内容一致**
- [ ] 所有图片引用为相对路径
- [ ] 文章有3层结构（热点/深度/数据）

**权力：** 不参与任何内容生产，仅执行质检。发现问题直接驳回。

### C-08 发布工程师
**输入：** 通过质检的全部产出
**输出：** 已发布的 GitHub Pages 网站

**流程：**
1. `git add <修改文件>`
2. `git commit -m "feat|fix: 扬说财经{版次} {日期} — 简明的改动说明"`
3. `git push origin main`
4. 验证: 请求网站URL确认可访问

### D-09 复盘分析师
**输入：** 当日全部产出 + 质量报告
**输出：** `docs/review/daily-{YYYY-MM-DD}.md`

**内容：**
1. 当日生产总结（产出物清单）
2. 问题记录（类型/严重度/处理方式）
3. 改进项（落实到下轮生产的清单）

## 用户确认门禁（新增v4.0强制）

### 适用场景

以下变更**必须先展示预览/设计稿，用户确认后才能执行**：

| 场景 | 预览方式 | 确认方式 |
|------|---------|---------|
| 文章结构重大调整 | 展示层级变化 | 用户确认 |
| SVG风格变更 | 展示新SVG截图 | 用户确认 |
| 新增/删除内容类型 | 展示新结构 | 用户确认 |
| 口播风格调整 | 展示脚本样例 | 用户确认 |

### 执行流程

```
[改动前] → [展示预览] → [等待用户确认] → [执行修改] → [再次展示] → [部署]
                                     ↑                    （如果用户不满意则回到修改）
                                  [用户拒绝则停止]
```

### 禁止场景
- ❌ 用户未确认的情况下直接推送重大变更
- ❌ 用户重复表达同一问题超过1次仍未理解（应停下来追问澄清）

## Claude Code ↔ Codex 协作协议（新增v4.0）

### 分工矩阵

| 任务类型 | 执行者 | 原因 |
|---------|--------|------|
| 内容叙事/文章框架/洞察写作 | Claude Code | 推理能力强，叙事质量高 |
| SVG代码生成 | Codex CLI | 代码量大，Codex批量生成快 |
| HTML/CSS结构搭建 | Claude Code | 需要理解整体设计一致性 |
| 文件操作（批量创建/移动/删除） | Codex CLI | 速度快，适合批量操作 |
| 质量审查/事实核查 | Claude Code | 需要深度推理 |
| 口播稿风格审核 | Claude Code | 需要理解人格化表达 |
| 数据搜索/采集 | Claude Code (WebSearch) | Codex无WebSearch能力 |
| 音频合成 | 本地命令 | edge-tts直接执行 |

### 协作流程

```
[Claude Code] 设计规划 → 产出设计稿/预览
       ↓
[Codex CLI] 批量执行 → SVG生成/文件操作
       ↓
[Claude Code] 质量审查 → 事实核查/风格检查/合规检查
       ↓
[用户确认]  → 预览展示 → 用户确认通过
       ↓
[Claude Code] 部署发布
```

### Codex执行规范

**委托Codex时，Claude Code必须提供：**
1. **精确的SVG设计描述** — 视觉隐喻、布局、颜色（非模糊要求）
2. **输出文件路径** — 绝对路径
3. **尺寸/格式约束** — viewBox、文件大小限制
4. **审查清单** — Codex产出后Claude Code按清单逐条审查

### 风险控制

- Codex生成的SVG必须在提交前经过Claude Code审查
- Codex执行批量文件删除时必须先列出被删除文件清单，经确认后执行
- Codex的DeepSeek模型可能"自信地猜" — Claude Code需要对其输出保持审慎

| 场景 | 操作 | 升级至 |

| **用户连续2次指出同一问题** | **停止并追问澄清** | **用户直接** |
