# 扬说财经 · 团队架构 v5.0

> Claude为质量门，Codex为生产力
> 版本: v5.3 | 更新: 2026-05-25
> 核心理念：产品质量是我们的核心竞争力，其他都是锦上添花

---

## 一句话哲学

**不管谁做，必须经过谁审。**

我们不再规定哪个模型必须做哪件事，只规定：任何产出在流向用户之前，必须经过哪道质量门。

## 质量门禁体系

这是我们唯一严格规定的东西——**质量门禁不可绕过**：

```
Layer 1: 事实核查门禁
  谁做: Codex 或 Claude Code
  审什么: 每个数据点必须有来源 URL，无来源的数据不能发表
  阻塞条件: 发现编造数据 → 整篇退回
  来源: 复盘 P0-01 "编造数据" 的教训

Layer 2: 风格一致性门禁
  谁做: Claude Code（唯一的）
  审什么: 文章风格是否匹配风格样本，SVG 是否符合素描风格
  阻塞条件: 风格偏离 → 退回修改，不可放行

Layer 3: 质量检查门禁
  谁做: Claude Code（唯一的）
  审什么: 6故事完整? 数据+叙事+洞察俱全? 合规声明存在?
          无套话？无占位符残留？音频时长匹配标注？
  阻塞条件: 任何一项不通过 → 不发布

Layer 4: 数据准确性门禁
  谁做: Claude Code（唯一的）
  审什么: 所有数据可回溯来源、无"历史第X次"等可疑表述、
          无模型内部知识替代实时搜索
  阻塞条件: 发现不可回溯的数据 → 不发布
```

## 谁做什么——指南不是铁律

以下是指南，不是铁律。核心判断标准是：**谁做这件事能让质量最高？**

| 工作 | 通常谁做 | 质量门谁审 | 说明 |
|------|---------|-----------|------|
| 新闻采集 | Codex（速度快） | Claude Code | Codex采集后Claude确认关键数据 |
| 事实核查 | Codex（上下文大） | Claude Code | Codex查来源→Claude复核 |
| 文章初稿 | Codex（效率高） | Claude Code | Codex按风格样本写→Claude修改润色 |
| 风格定调/样板 | Claude Code | — | Claude写第一篇作为风格参照 |
| 漫画故事板设计 | Claude Code | Claude Code | 设计视觉隐喻、构图、角色出场 |
| 高品质SVG生成 | Claude Code（核心） | Claude Code | 全彩渐变背景、多层构图、阿扬IP角色贯穿 |
| SVG增强（标题栏等） | upgrade-comic.py | Claude Code | 添加叙事标题栏、投影、品牌装饰 |
| 口播稿 | Codex（按模板） | Claude Code | Codex写初稿→Claude人格化润色 |
| 音频合成 | edge-tts | 文件大小验证 | 纯机械 |
| 数据图表 | generate-charts.py | 检查生成成功 | 纯机械 |
| 质量终审 | Claude Code | — | 唯一把关人 |
| 复盘改进 | Claude Code | — | 需追溯根因 |

**关键原则：** 以上"通常谁做"可以灵活调整。比如Claude Code状态好时可以直接写文章，Codex空闲时可以做更多审核辅助。**灵活性以质量为前提：任何调整都必须通过质量门禁。**

## 来自教训的铁律（13条）

以下每一条都是付过学费的，**永不再犯**：

### 铁律1: 数据不准，整篇重做
- 来源: P0-01 编造数据事件（5.22早报写"3.5万亿成交量是A股历史第二次"）
- 规则: 每个百分比/价格/历史对比必须有精确来源URL
- 检查: 终审时必须逐条追溯到来源，追溯不到的删掉

### 铁律2: 标多少就做多少
- 来源: P0-02 音频标注15min实际5min
- 规则: 音频时长、字数、阅读时间——标多少就是多少
- 检查: 终审时验证脚本字数→预计时长→实际时长三者一致

### 铁律3: 风格不可随意变更
- 来源: 
  - 5.24测试日报使用暗黑风格（废弃已有风格）
  - 5.24同日漫画擅自改成深色渐变背景+加入角色形象（废弃白底线稿风格）
- 规则: 
  - 每次生产前先读风格样本（漫画看5.22日报，文章看央行法修订专题）
  - 风格变更（背景色/尺寸/配色/新增元素）必须先展示预览，用户确认
- 检查: 终审时与风格样本逐项比对

### 铁律4: 模板占位符不能残留
- 来源: P3-09 专题模板`{{AUDIO_TIME}}`未替换
- 规则: HTML中不能出现`{{`字符（CSS content除外）
- 检查: 发布前grep检查

### 铁律5: 用户反馈不能忽视
- 来源: 用户多次纠正同一方向问题
- 规则: 用户连续2次指出同一方向错误 → 停下来追问，不再猜测
- 检查: Claude Code终审时自问"这个方向用户之前有没有说过什么？"

### 铁律6: 未确认不新增元素
- 来源: 5.24漫画擅自将阿扬IP形象加入每张漫画（用户未要求）
- 规则: 任何新增内容类型/视觉元素/功能模块，必须先征求用户意见
- 检查: "用户有没有说过要做这个？"→ 没有就别做

### 铁律7: 使用已有工具链
- 来源: 5.24生产全流程未运行 `generate-charts.py`（已有脚本被闲置）
- 规则: 现有Python脚本（generate-charts.py等）是标准生产流程的一环，不能跳过
- 检查: 终审时确认 `docs/charts/` 目录当天有更新

### 铁律8: 用户同问题≥2次 → 冻结专攻
- 来源: 5.25早报"首页不显示"同一问题被用户指出5次才真正解决
- 规则: 用户连续2次指出同一方向错误 → 暂停所有其他工作，专攻此问题直到解决。不允许带病推进。
- 检查: 终审时自问"这个问题用户之前说过几次？"

### 铁律9: JS变更必须浏览器验证
- 来源: 5.25早报 `data[s]` 应为 `data[s.key]` 的JS bug，在未验证的情况下直接推送了3轮
- 规则: 所有前端JavaScript变更在推送前必须通过浏览器运行/控制台调试确认正确，不得"先推了再说"
- 检查: 修改JS文件后必须在Chrome DevTools中执行关键路径代码，确认无报错且渲染正确

### 铁律10: 推送前必须部署验证
- 来源: 5.25早报出现多次"源码有但页面不显示"、"音频没更新到网站"等部署问题
- 规则: 
  - 每次git push后、通知用户前，必须执行验证：
    ```
    □ curl index.html | grep data-date → 确认 data-date 为当日
    □ curl articles.js → 确认 latest_date + 当日内容存在
    □ curl article.html → HTTP 200 且 内容正确
    □ curl audio.mp3 | 对比文件大小 → 确认是正确版本
    ```
  - 任一验证不通过 → 不通知用户，先修复
- 检查: 通知用户"已部署"之前，逐项过验证清单

### 铁律11: 引用参照物，不依赖记忆
- **来源:** 5.25晚报漫画连续2轮风格错误（先文字堆砌、后深色背景），均未读取5.22已有风格样本
- **规则:**
  - 创建/修改任何视觉内容前，**必须先读取对应的风格参照文件**，现场确认后再生产
  - 禁止"我记得上次的风格"——记忆不可靠，必须现场读文件
  - 风格参照映射：
    ```
    晚报漫画 → 先读 5.22 comic/panel-001.svg 到 panel-006.svg
    文章模板 → 先读 5.22/5.25 article.html
    专题漫画 → 先读 special/央行法修订/comic/panel-*.svg
    ```
- **检查:** 生产漫画/HTML前，必须有一个Read文件操作确认参照

### 铁律12: 路径变更必须 curl 验证
- **来源:** 5.25晚报返回首页链接从正确`../../../index.html`改成错误`../../../../index.html`，未验证路径解析
- **规则:**
  - 任何相对路径（href/src/链接）的修改，推送前必须用 curl 确认可访问
  - 路径解析方法：从文章URL手动计算层级数，然后用 `curl -sI 完整URL`
  - 示例：`curl -sI https://jayyuziyang-lang.github.io/yzy/2026-05-25/../../../../index.html`
- **检查:** 路径修改后不curl验证 → 不准推送

### 铁律13: 修改不等于推送，中间有验证环节
- **来源:** 5.25全天所有问题根因：改了就直接推送，没有"改完→验证→确认无误→推送"的中间环节
- **规则:**
  - 每轮修改后必须经过验证才能推送，验证不得少于：
    ```
    □ 如果是SVG → Read确认内容正确（不看就推 = 违规）
    □ 如果是HTML → grep确认关键标签存在
    □ 如果是JS → 手动推导输入输出或浏览器测试
    □ 如果是路径 → curl确认可访问
    □ 如果是数据 → 与实际来源对比
    ```
  - "先推了再看结果" = 严重违规
  - 验证不通过的东西不准进入 git commit
- **检查:** commit message中必须体现验证方式（如"已验证：curl 200 + Read确认"）

### 铁律14: 部署必须使用 deploy.sh
- **来源:** 5.25全天反复出现"改完HTML忘记run update-index.py"、"改完数据手动git push不验证"等部署流程遗漏
- **规则:**
  - `deploy.sh` 是 **唯一授权** 的部署方式
  - 自动执行: update-index.py → pre-deploy-check验证 → site-health → git push → CDN预热
  - 严禁手动 `git add . + git commit + git push` 绕过
  - 部署命令: `bash deploy.sh "提交信息"`
- **检查:** 所有提交必须通过 deploy.sh。发现手动git push = 严重违规

## 🚨 LLM偷懒对抗机制（Agent Guard v1.0）

> 以下机制专门应对AI模型的"过早满足"倾向——推了再说、差不多就行、不验证就交付。

### 问题诊断

AI模型（包括Claude Code）在生产中常见以下偷懒行为：

| 行为 | 表现 | 5.25实例 |
|------|------|---------|
| 不读参照就做 | 凭记忆做风格判断 | 漫画连续2轮风格偏离 |
| 不验证就推 | 改完直接commit | 全部9个问题 |
| 不核实就复制 | 直接复制旧文件不改内容 | panel-001用5.22美伊内容 |
| 不调试就猜 | 从现象直接跳到结论 | JS bug猜了4轮 |
| 满足于"看起来对" | 不深究细节 | 返回首页路径多算一层 |

### 启动前必过核查（三问）

在执行任何生产动作之前，先回答以下三个问题：

```
Q1: 我要改什么？→ 具体到文件名、行号、改动内容
    （答不出 → 先读，不要动手）

Q2: 我怎么验证改对了？→ 用哪个命令/哪个URL/哪种方法
    （答不出 → 先设计验证方案，再动手）

Q3: 历史上/参照里的正确做法是什么？→ 现场读参照文件确认
    （答不出 → 先读参照，不要凭记忆）
```

**三问答不全 → 不能开始执行。**

### 质量越级机制

当问题重复出现时，触发自动升级：

```
用户第1次指出 → 记录到反馈
用户第2次指出同类问题 → 冻结当前工作，追根因
    ↓ 根因写入铁律 / 流程 / 模板
用户第3次指出同类问题 → 降级模型：更换执行模型
    ↓ 或切换执行方式（如用Codex替代Claude执行）
用户第4次指出同类问题 → 重构工作流：
    → 质疑当前流程设计是否有效
    → 是否需要引入自动化检查脚本
    → 是否需要人工介入
```

**连续3次同方向错误 = 当前的执行者/流程不适合这个任务。**

### 部署规范：deploy.sh 是唯一通道

**禁止手动 git push。** 所有部署必须通过：

```bash
bash deploy.sh "提交信息"
```

`deploy.sh` 自动执行：
1. `update-index.py` — 刷新 articles.js/data
2. `pre-deploy-check.sh` — 6项验证（含预渲染、链接、占位符）
3. 验证通过 → 自动 git add/commit/push
4. CDN预热 + 部署后验证

手动 git push = 跳过所有验证环节 = 严重违规。

### 每一次推送前的自检命令

在 `git push` 之前（如因特殊原因无法使用 deploy.sh），强制执行：

```bash
# 自检1: 我改了哪些文件？
git diff --stat

# 自检2: 每个改动我能证明它是正确的吗？
# 如果不能，停下，不要推送。

# 自检3: 改动会不会影响其他功能？
# 比如改article.html → 会影响update-index.py读取的title吗？

# 自检4: 我读了所有相关的参照文件吗？
# 没有读 → 回去读。
```

### 风格确认清单（漫画生产前强制）

```
□ 已读取5.22晚报漫画作为风格参照
□ 已确认背景色 = #FAFAFA（浅色，非深色）
□ 已确认不存在暗色背景/渐变背景
□ 已确认以几何图形为主，不以文字堆砌为主
□ 每张漫画的关键数字 ≤ 4个
□ 标注文字 ≤ 2行
□ 文件大小 < 6KB
```
