## 一、定位与使命

产业链深度分析是**扬说财经专业性的核心体现**。与日报不同，产业链文章不追求"快"，而追求"深"——目的是让读者在读完一篇文章后，**对一个产业的理解追上专业从业者的水平**。

我们的目标读者画像：
- **主力读者**：有财经基础但不具备电子/材料/半导体专业知识的投资者
- **次级读者**：产业从业者和专业分析师（他们关注数据准确性和逻辑严密性）

写作原则：**让非专业人士看懂，让专业人士觉得有料。**

## 二、质量标准总纲（MLCC文章为模板）

以下标准以本次MLCC文章为具体范例，每一条均可从该文中找到对应示例。

### 2.1 叙述标准

#### A. 三层叙述法（强制）

每篇文章的大章节应该遵循**三层结构**，由浅入深：

| 层 | 目标 | 方法 | MLCC示例 |
|----|------|------|----------|
| 第一层：直觉理解 | 让外行"看个热闹" | 生活类比、思想实验 | "电容是微型水库""MLCC是千层蛋糕""电RAM"类比 |
| 第二层：逻辑推演 | 让内行"看个门道" | 因果链、数据驱动 | GPU功耗↑→电流↑→纹波↑→MLCC↑ 传导链 |
| 第三层：底层规律 | 让所有人"带走认知" | 提炼不可逆的结构性规律 | "高端MLCC定价权在技术方不在客户"、"时间壁垒是真正的护城河" |

**检查要点：** 每个章节必须至少包含一个生活类比和一个底层规律提炼。

#### B. 引入段法（参考§4写法）

每个核心概念引入时，遵循以下序列：
1. **思想实验**或**生活场景**（"想象你在开一辆电动车……"）
2. 用熟悉的系统拆解成三部分（PSU=自来水厂、VRM=楼顶水箱、MLCC=水龙头储水罐）
3. 揭晓结论，引用数据佐证

#### C. 图表深度分析法（重要）

**禁止做法：** "下图显示了X数据"——这是描述，不是分析。

**强制做法：** 每个主要图表后必须包含以下结构：

```
段落1：这张图揭示了什么结构性问题？（不仅描述，要提炼）
段落2：为什么是这个数字？底层驱动力是什么？
段落3（可选）：这意味着什么？对产业链各环节有什么不同影响？
```

MLCC文章中的示例（§5市占率图分析）：
- 段落1提炼："市场分两层，两层是不同游戏"
- 段落2分析根本原因："时间复利"——村田80年积累
- 段落3推演影响："时间壁垒才是真正的护城河"

### 2.2 排版标准

#### 文字层次（6级视觉系统）

| 元素 | 类名 | 用途 |
|------|------|------|
| 大标题 | `h1.article-title` | 文章标题，40px/800字重 |
| 章节标题 | `h2.section-title` + `.section-number` | 章节标题，30px/编号圆标 |
| 子标题 | `h3.sub-title` | 章节内子主题，20px/700字重 |
| 引导段 | `p.lead` | 章节开篇，左蓝色条+浅蓝背景 |
| 正文 | `p` | 正文段落，16px/行高1.85 |
| 文本高亮 | `.highlight` / `.key-point` | 关键数据/核心论断的高亮 |

#### 排版组件列表

| 组件 | 类名 | 来源参考 |
|------|------|---------|
| 关键数据卡片 | `.data-grid` + `.data-card` | 三列数值展示 |
| 核心判断卡片 | `.insight-card`（蓝/绿/琥珀） | 左色条+浅背景 |
| 引用块 | `.pull-quote` | 上下边线，居中 |
| 章节分割线 | `.section-divider` | 渐变色分割线 |
| 行内标签 | `.tag-blue` | 小标签，白字蓝底 |
| 双列卡片 | `.card-grid` | 并列对比 |
| 术语表 | `.glossary` | 专业术语解释 |
| 周期标注 | `.cycle-note` | 特别说明（琥珀色） |

**禁止行为：** 禁止在HTML中使用内联style（如 `style="font-size:20px;..."`）控制排版——所有样式必须通过CSS类实现。

### 2.3 语言标准

| 要求 | 做法 | 反面案例 |
|------|------|---------|
| 使用生活类比 | "电容=微型水库""MLCC=千层蛋糕""供电系统=大楼供水" | 纯技术术语堆砌 |
| 明确因果链 | "GPU功耗↑→电流↑→纹波↑→MLCC↑" | "需求增长推动MLCC用量增加" |
| 数据带入场景 | "从200颗到5000颗，不是线性增长是指数爆发" | "MLCC用量增长较快" |
| 提炼结构性规律 | "定价权由技术壁垒决定，不在客户手里" | "高端MLCC毛利较高" |
| 每章收尾投资视角 | 用`📌 投资视角：所以呢？`或洞察卡收尾 | 章节结束无总结 |

### 2.4 内容完整性清单

每篇产业链文章必须包含以下板块：

- [ ] **引语+核心问题**（文首，说明"为什么是现在谈这个"）
- [ ] **物理/技术基础**（用类比解释，1-2章）
- [ ] **核心逻辑推演**（最核心的因果链展开，1-2章）
- [ ] **竞争格局**（分层次分析，含数据图表）
- [ ] **市场与周期**（区分周期性与结构性）
- [ ] **趋势展望**（未来12-24个月关键变量）
- [ ] **全文总结**（三件事带走）
- [ ] **术语表**（小白友好，白话解释）
- [ ] **数据来源**（可追溯，无来源不发表）
- [ ] **免责声明**（产业科普，非投资建议）

## 三、今日复盘（2026-06-12）

本日MLCC文章生产过程中暴露了多个系统性问题，逐条复盘：

### 问题1：视频播放卡顿

**现象：** 月6MB的视频从GitHub Pages直接加载，在无国内CDN节点的情况下播放严重卡顿。

**根因：** 没有在文章生产初期就考虑视频传输路径和压缩策略。

**整改措施：**
1. 视频必须ffmpeg压缩到1MB以下（CRF 28, 预设fast, 720p）
2. 必须准备jsDelivr CDN源作为首选播放来源
3. 封面图（poster）必须提取视频首帧，用JPG格式（<50KB）
4. preload设为auto（用户明确否认懒加载方案）
5. 以上4条已写入此文档§4.6

### 问题2：CSS标签未闭合导致文章全页面崩溃

**现象：** `fix-all-issues.py`替换style块时缺少`</style>`，导致浏览器将整个HTML视为CSS，文章完全不可渲染。

**根因：** 脚本生成的替换字符串缺少闭合标签，审查时未逐行验证可用性。

**整改措施：**
1. 任何对HTML的脚本化修改都必须执行"修改→打开验证"循环
2. 关键标签（`<style>`、`<div>`、`<script>`）的闭合性必须在diff中检查
3. site-health.py增加了页面内容的可用性验证（返回200 + 包含关键内容）

### 问题3：排版层次不清晰，非专业人士理解困难

**现象：** 原文排版使用大量内联style，缺乏视觉层次；图表描述浮于表面，未进行逻辑深度分析。

**根因：** 没有建立产业链文章的排版和叙述标准。

**整改措施：**
1. 创建此文档作为所有产业链文章的生产标准
2. 从MLCC文章中提取了6级视觉层次系统和排版组件库
3. 制定了三层叙述法（直觉→推演→规律）
4. 制定了图表深度分析法（禁止纯描述，强制提炼+推演）

### 问题4：deploy.sh rebase冲突

**现象：** `data/latest-cdn.json`在每次部署时都会产生合并冲突，因为deploy.sh在CI管道中同时被多个操作修改。

**现状：** deploy.sh的自动rebase+接受本地版本方案工作正常（预部署验证全部56项通过），但每一次冲突都意味着多花几秒重跑。暂时可用，后续应考虑将latest-cdn.json纳入.gitignore或改用更优雅的冲突解决策略。

## 四、团队协作规范

### 4.1 分工矩阵（产业链文章适用）

| 工序 | 执行 | 质量门 | 时间 |
|------|------|--------|------|
| 选题+定框架 | Claude定调 | — | 2h |

| 部署上线 | Claude Code | deploy.sh全流程通过 | 10min |

### 4.2 质量门禁（不可绕过）

1. **数据准确性门禁：** 每个数据点必须标注来源，无来源不发表
2. **叙述完整性门禁：** 对照§2.1三层叙述法检查每章是否达标
3. **排版一致性门禁：** 对照§2.2禁止内联style，必须使用CSS类
4. **图表深度分析门禁：** 每个主要图表后必须有"提炼+推演"段落
5. **可理解性门禁：** 至少一位非专业背景人员通读后能理解核心论点

### 4.3 文档优先原则

本文件（`CHAIN_ANALYSIS_STANDARD.md`）是产业链系列文章生产的最高参考。生产任何产业链文章前，生产方必须：
1. 重新读取§2质量标准总纲
2. 用MLCC文章作为视觉和风格参考打开
3. 生产完成后对照§2.4内容完整性清单逐项自查

### 4.4 后续迭代

- 本文件每完成一篇新的产业链文章后更新
- 新文章中发现的任何新质量问题需补充到§3
- 新文章中产生的更优写法需补充到§2
- 本文件版本历史见文末
