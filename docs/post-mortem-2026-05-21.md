```
Why 1: 改完就直接 push 到 main 分支

      → Why 4: 没有"先测试再上线"的分支策略
```

**结论：** **main 分支和生产环境耦合**，任何推送都立即上线，没有缓冲期。


## 六、经验教训（知识库）

### L1: 永远不要在 main 分支上直接改代码
**教训：** 没有预览环境的直接改动，等于在生产环境做实验。
**对策：** 所有改动先在本地验证，`git diff` 审查后再 push。

### L2: "沉默的错误"比"明显的错误"更危险
**教训：** `2>/dev/null` 把失败变成了静默跳过，问题潜伏了 3 轮才发现。
**对策：** 脚本中永远不要静默吞掉错误。如果必须处理错误，写明确的 `|| handle_error`。

### L3: 静态页面也要考虑"JavaScript 不可用"的情况
**教训：** 100% JS 依赖让 Google Bot 也看不到你的内容。
**对策：** 关键内容（文章列表、导航）必须在 HTML 中可见，JS 只做增强。

### L4: CDN 缓存是"隐形的时间机器"
**教训：** 部署后看到的不是最新版本，而是 CDN 缓存的版本。
**对策：** 部署脚本必须做：轮询等待 → 验证关键 URL → 预热 CDN 缓存 → 通知用户。

### L5: 改动后必须按"完整用户路径"走一遍
**教训：** 改完 JS 渲染逻辑，只看代码没发现问题，但实际页面已经坏了。
**对策：** 每次部署后用 `scripts/verify-deploy.sh` 按用户视角验证 5 个关键页面。
